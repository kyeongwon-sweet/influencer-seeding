# 설계안: 시트↔DB 양방향 → DB 단일 정본(단방향) 전환

작성: Claude, 2026-07-14. **제안서 — 실행 전 사용자+Codex+시트세션 합의 필요**(워크플로 변경 포함).

## 왜 (근본원인 요약)
- 시트는 **위치 기반**(열=날짜, 행=게시물) + **수기 편집**. 값이 "어느 게시물 것"인지 URL로 검증 안 함.
- **매일 양방향 동기화**(importStats 시트→DB, exportStats DB→시트)가 돌며, 날짜열 중복/어긋남·수기 드래그가 만든 오염이 **왕복마다 번지고 compound**.
- 결과: 미러링/종료 게시물 다수의 조회수가 뒤섞임(부분일치 37~74%, 한 행에 여러 게시물 값 교차). 셀 단위 자동 정리 불가.

## 목표 구조 (단방향)
```
수집 소스        →   DB (단일 정본, post_id+date 키)   →   시트 (표시 전용)
- run_monitoring     - post_daily_stats                    - exportStats(DB→시트)만
- 대시보드 수기입력   (키 기반이라 위치 어긋남에 강함)        - 사람이 시트에 입력해도 DB로 안 감
```
핵심: **시트→DB 경로(importStats)를 제거**한다. 시트는 DB를 비추는 거울(읽기)일 뿐, 정본이 아니다. 수기 입력은 **대시보드**에서만(이미 DB로 저장됨).

## 마이그레이션 순서 (안전)
1. **DB 정리(키 기반이라 안전)**: 오염 게시물별로 행 전체 삭제 → 실측(재수집) 최종값 1행 삽입(= B 최소정정). Claude 수행. 백업 필수.
2. **importStats 차단** 배포(A). 이후 시트 수기가 DB로 안 감.
3. **시트 데이터영역(날짜 열) 클리어 → exportStats 재실행** → 깨끗한 DB로 시트 재생성(엉킨 이력 통째 대체). 시트세션 수행, 백업 필수.
4. **정기 스캔**(`scripts/scan_cross_post_copies.py`) 주1회로 감시.
> ⚠️ 순서 중요: DB부터 깨끗이(1) → import 끊고(2) → 시트 재생성(3). 순서 어기면 오염이 다시 섞임.

## 워크플로 영향 (팀이 알아야 할 것)
- **더 이상 연동 시트에 조회수를 직접 입력하지 않습니다.** 수기 입력·정정은 **대시보드**에서. (시트에 적어도 DB에 반영 안 됨.)
- 시트는 **보고/열람용 스냅샷**(DB가 매일 채워줌). 시트에서 고쳐도 다음 exportStats에 덮일 수 있음.
- 카카오처럼 Apify 수집 불가 지표도 **대시보드 수기 입력**으로.

## 경량 대안 (양방향 유지가 꼭 필요하면)
importStats를 남기되: (a) **URL(게시물 키)로만 매칭**(위치 매칭 폐기), (b) **중복 날짜열 제거·정렬 고정**, (c) 복사-가드 유지. → 구현 복잡·여전히 수기 취약. **비권장**(단방향이 근본적).
